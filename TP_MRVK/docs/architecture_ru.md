# Архитектура Проекта: Навигация В Динамической Среде

## 1. Модуль Ввода Данных (Input Module)

Задача модуля: собрать и синхронизировать сенсорные данные до стадии построения карты.

Поток данных:

1. Лидар (3D или 2D) дает облако/скан препятствий.
2. Система позиционирования (одометрия, EKF, RTK GPS) дает позу робота.
3. Данные синхронизируются по времени и приводятся к общей системе координат.

Для 3D-лидара применяется высотный срез:

- отбрасывается земля ниже порога $z_{min}$
- отбрасываются точки выше высоты робота $z_{max}$
- оставшиеся точки проецируются на плоскость $(x, y)$

Результат: компактное 2D-представление окружения для быстрого обновления карт и планирования.

Связь с текущим кодом:

- предобработка скана для SLAM/навигации реализована в `mrvk_gazebo/scripts/scan_static_temporal_filter.py`
- основной рабочий контур запускается без `scan_cleaner_node`

## 2. Модуль Построения Карт (Mapping Module, Ядро)

Вместо распознавания классов объектов напрямую оценивается, как меняется свободное пространство.

### 2.1 ESDF-карта

Для каждой ячейки сетки хранится расстояние до ближайшего препятствия:

$$
ESDF_t(x, y) = \min_{(i, j) \in O_t} \sqrt{(x-i)^2 + (y-j)^2}
$$

где $O_t$ — множество занятых ячеек в момент времени $t$.

### 2.2 Дифференциальная ESDF-карта (Delta ESDF)

Ключевая идея:

$$
\Delta ESDF_t(x, y) = \frac{ESDF_t(x, y) - ESDF_{t-1}(x, y)}{\Delta t}
$$

Интерпретация знака:

- $\Delta ESDF < 0$: препятствие приближается к данной зоне
- $\Delta ESDF > 0$: препятствие удаляется
- $|\Delta ESDF|$: интенсивность изменения (косвенная скорость/угроза)

### 2.3 Динамическое Обновление Occupancy (insertQueue/deleteQueue)

Чтобы не оставлять ложные "стены" за движущимися объектами, локальная карта обновляется при каждом шаге:

1. `insertQueue`: ячейки, которые стали занятыми
2. `deleteQueue`: ячейки, которые освободились

Псевдологика шага $t \rightarrow t+1$:

1. Добавить новые занятые ячейки в `insertQueue`
2. Добавить освобожденные ячейки в `deleteQueue`
3. Применить обе очереди к occupancy-сетке
4. Пересчитать ESDF
5. Вычислить $\Delta ESDF$

Результат: динамические объекты удерживаются в памяти ровно столько, сколько нужно для локального прогноза, затем корректно "забываются".

## 3. Модуль Планирования (Planning Module)

Состоит из двух уровней: глобального и локального.

### 3.1 Глобальный Планировщик

- строит маршрут на статической карте от старта к цели
- базовый алгоритм: A*
- выход: набор опорных точек (waypoints)

Связь с текущим проектом:

- глобальный планировщик задается в `mrvk_gazebo/config/navigation/move_base.yaml`
- при `use_dijkstra: false` в `global_planner/GlobalPlanner` используется режим A*

### 3.2 Локальный Планировщик

Локальный уровень отвечает за "здесь и сейчас": безопасный проезд среди статических и динамических препятствий.

Базовый цикл:

1. Генерация набора коротких кандидатных траекторий
2. Отсев траекторий, проходящих через статические препятствия
3. Оценка оставшихся траекторий по критериям $S1, S2, S3$
4. Выбор траектории с максимальным итоговым баллом

Целевая схема оценки:

$$
Score = w_1 S1 + w_2 S2 + w_3 S3
$$

где:

- $S1$ (направление): насколько траектория ведет к следующей глобальной точке
- $S2$ (статика): запас до статических препятствий (по ESDF)
- $S3$ (динамика): безопасность относительно динамики (по $\Delta ESDF$)

Рекомендованные веса из постановки:

- $w_1 = 10$ (фокус на цель)
- $w_2 = 0.5$ (статика)
- $w_3 = 1$ (динамика)

Правило динамики:

- траектории, ведущие в области с $\Delta ESDF < 0$, штрафуются
- траектории в области с $\Delta ESDF \ge 0$, предпочитаются

Связь с текущим проектом:

- локальный улучшенный DWA реализован в `../improved_dwa_local_planner/src/improved_dwa_local_planner.cpp`
- динамический скоринг использует аналитический ESDF и Delta ESDF штраф/бонус
- источник динамических препятствий в рантайме: топик `/obstacles` от `obstacle_detector`

## 4. Взаимодействие Глобального И Локального Уровней

1. Глобальный уровень прокладывает оптимальный маршрут по статической карте.
2. Если на маршруте появляется пешеход/динамический объект, локальный уровень временно отклоняется от глобального пути.
3. После освобождения зоны локальный уровень возвращает робота на глобальные waypoints.

Это предотвращает "застревание" и позволяет держать стратегию движения стабильной даже при плотной динамике.

## 5. Рекомендуемая Декомпозиция Пакетов

Если требуется строгое разделение по модулям, удобно поддерживать следующую структуру:

1. `input_module`: синхронизация сенсоров, фильтры, 3D->2D проекция
2. `mapping_module`: occupancy, ESDF, Delta ESDF, очереди insert/delete
3. `planning_module_global`: A* и управление waypoint-маршрутом
4. `planning_module_local`: генерация/оценка траекторий, команды управления

В текущем репозитории часть функций уже распределена между:

- `mrvk_gazebo/scripts/scan_static_temporal_filter.py`
- `improved_dwa_local_planner`
- конфигурацией `move_base` в `mrvk_gazebo/config/navigation`

Это позволяет развивать архитектуру итеративно без полного переписывания существующих узлов.
